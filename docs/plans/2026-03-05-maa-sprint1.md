# MAA Sprint 1 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Scaffold the MAA repo with working `maa review frontend <project-path>` command that runs concrete bash checks and generates a markdown report.

**Architecture:** Local-first, bash-only. A top-level `maa` entrypoint routes `review frontend <path>` to `scripts/maa-review.sh`, which runs deterministic grep/test checks against the target project and renders a filled markdown report. All standards, policies, and skill docs are plain markdown. No Python, no Node, no database, no jq.

**Tech Stack:** Bash, Markdown, YAML, Git.

### Assumptions

- Tool name is `maa` (not `forge` as in design doc)
- Config dir is `.maa/` (not `.forge/`)
- Reports go to `reports/reviews/YYYY-MM-DD-<project>-frontend.md`
- `reports/reviews/` is created at runtime if missing
- No jq — JSON parsing done with grep/sed
- Uncertain checks marked "not detected" not "fail"
- No sub-agents needed; all tasks are file creation

### Checks Implemented in maa-review.sh

1. `package.json` exists → pass/fail
2. `src/` directory exists → pass/fail
3. `public/` directory exists → present/absent (not a failure)
4. Framework signal: grep for react/next/vite in package.json → detected/<name>/not detected
5. Linting signal: eslint config file or eslint in package.json → detected/not detected
6. Formatting signal: prettier config file or prettier in package.json → detected/not detected
7. Test signal: test script in package.json or jest/vitest/cypress config → detected/not detected
8. `README.md` exists → present/absent
9. `.gitignore` exists → present/absent
